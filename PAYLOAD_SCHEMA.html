Webhook payload posted to legacy-order-ingest on every trigger fire.

Source: Cloudflare Worker asana-shooter · Project: ShoeCare Master Board (gid 1202289964354061)

1. Trigger

The Worker fires this payload once per transition when the Asana custom field Assessment System on any task in the project is set to the value "New" (whitespace-tolerant).

Edge-triggered: re-firing requires the field to be cleared and re-set. Toggling unrelated fields on the same task does not re-fire.

5. Photo URLs

Every link in before_images_links and other_images_links points to Cloudflare R2 at https://pub-e253fc5bb5ec4740bf58f9cc062ed9b3.r2.dev. URLs are permanent (no expiry) — unlike Asana's signed download URLs which expire after ~10 days.

Path format: /tasks/<asana_task_gid>/<asana_attachment_gid>/<original_filename>. Filenames are sanitised (non-alphanumeric chars become _).

8. Operational notes

  • If validation on the destination function returns non-2xx, the Worker logs the error and the trigger state remains uncommitted in KV. The next Asana event for that task will retry the destination POST automatically.
  • Photo URL format depends on R2 public access via https://pub-e253fc5bb5ec4740bf58f9cc062ed9b3.r2.dev. If you switch to a custom domain later, all *_images_links values change — coordinate with Lovable beforehand.
  • The full notes field is included verbatim. Downstream may parse it for additional fields not promoted to the top level (e.g. Auto Approval, Shoe Assessment).
  • Garbage upstream values "None", "none", "N/A", "null", "undefined", and "-" are normalised to JSON null on every extracted field.
  • Task name segment 5 may contain either "Size: 44 EU" (older format) or just "44 EU" (newer format). The parser handles both. If neither is present in the name, shoe_size falls back to the notes Size: label.

9. Callback endpoint (Lovable → Worker)

After processing, Lovable POSTs back to the Worker so the approval URL can be appended to the Asana task's description.

MethodPOST
URLhttps://asana-shooter.washmen-egolf.workers.dev/lovable-callback
AuthorizationBearer <CALLBACK_SECRET> (shared secret; ask the Worker operator)
Content-Typeapplication/json

Request body

{
  "task_gid":     "1214429954181947",
  "approval_url": "https://service-wizard-kit.lovable.app/approved/f7680eeb-3deb-493f-86a1-4d361a9d5038"
}
FieldTypeNotes
task_gidstring (digits)Asana task identifier. Echo back the value Lovable received in the original payload (task_gid).
approval_urlstring (https URL)The Lovable approval link to surface on the Asana task.

Responses

StatusMeaning
202Accepted; the Worker queued the Asana update. Idempotent — re-posting the same URL is a no-op.
400Missing or malformed body (invalid JSON, missing fields, non-https URL, non-numeric task_gid).
401Missing or wrong Authorization header.

What the Worker does on success

  1. Fetches the current task notes from Asana.
  2. If the URL is already in the notes, exits (idempotent).
  3. Otherwise appends a line: Customer approval response: <approval_url> to the description and PUTs the updated notes back to Asana.
